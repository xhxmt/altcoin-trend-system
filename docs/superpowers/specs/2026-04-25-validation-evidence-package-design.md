# Validation Evidence Package Design

Date: 2026-04-25
Version: v1.0

## Context

The validator has recently gained the core trust semantics needed for v2 signal validation: explicit signal timing, availability-based forward scans, coverage-aware summaries, prepared-frame integration tests, optional DB smoke coverage, and 30d/90d comparison policy checks. The next step is not another threshold adjustment. The next step is to produce a real, reproducible evidence package from the current validator and local market data.

This design covers a P0 evidence-package workflow only. It intentionally separates evidence generation from later P1 hardening work such as data-health metadata, no-lookahead regressions, and shadow rollout automation.

## Goal

Create a repeatable evidence runner that produces a local validation evidence package for the current v2 signal outputs. The package must show whether the validator, real DB path, selector artifacts, and optional comparison artifacts are strong enough to support threshold-change decisions.

The workflow must preserve the current validator's semantics. It orchestrates validation runs and summarizes artifacts; it does not change signal rules, threshold logic, forward-label definitions, or coverage policy.

## Scope

In scope:

- Generate a local evidence package under `artifacts/autoresearch/validation/<YYYY-MM-DD>/`.
- Run the targeted validation test suite and impacted signal/backtest suite.
- Run the optional real-DB smoke test as a required evidence gate.
- Compute a fixed DB-aware `end_at` for the package.
- Run 30-day validation artifacts for all current output selectors:
  - `continuation`
  - `ignition`
  - `ignition_A`
  - `ignition_B`
  - `reacceleration`
  - `reacceleration_A`
  - `reacceleration_B`
  - `ultra_high_conviction`
- Preserve each selector's native validator artifacts: `summary.json`, `metadata.json`, `signals.csv`, and `README.md`.
- Run before/after comparison only when traceable baseline and candidate config files already exist.
- Generate a machine-readable `run_manifest.json` and human-readable `EVIDENCE_PACKAGE.md`.

Out of scope:

- Threshold tuning.
- New signal families.
- Synthetic or inferred baseline/candidate config construction.
- Writing one-off validation results into `docs/strategy/current-strategy.md`.
- Data-health field expansion.
- No-lookahead regression tests.
- Shadow rollout scheduling.
- Full backfill quality repair.

## Artifact Layout

The evidence package directory should use this structure:

```text
artifacts/autoresearch/validation/<YYYY-MM-DD>/
  EVIDENCE_PACKAGE.md
  run_manifest.json
  test_logs/
  db_smoke/
  selectors/
    continuation/30d/
    ignition/30d/
    ignition_A/30d/
    ignition_B/30d/
    reacceleration/30d/
    reacceleration_A/30d/
    reacceleration_B/30d/
    ultra_high_conviction/30d/
  comparisons/
```

Each selector directory must contain the artifact files generated by `scripts/validate_ultra_signal_production.py`. The runner may invoke the validator with a temporary output root and then move or copy the single generated run directory into the selector path, but the final package must make each selector's 30-day artifact easy to find.

## Evidence Runner Interface

Add a thin orchestration entrypoint:

```bash
.venv/bin/python scripts/run_validation_evidence_package.py \
  --output-root artifacts/autoresearch/validation \
  --window-days 30
```

Supported options:

```text
--end-at <iso timestamp>       Manually override the DB-aware end_at.
--selectors <comma list>       Override the default selector set.
--comparison-root <path>       Search for traceable baseline/candidate comparison configs.
--skip-tests                   Resume/debug mode only; not valid for a passing evidence gate.
```

The runner is an orchestration layer only. It should use subprocess calls to existing commands where practical and should not duplicate the validator's label, coverage, selector, or comparison logic.

## Execution Flow

1. Create the package directory and initialize a partial `run_manifest.json`.
2. Record git SHA and whether the worktree is dirty.
3. Run the targeted validation tests:

   ```bash
   .venv/bin/pytest \
     tests/test_validate_signal_semantics.py \
     tests/test_validate_ultra_signal_production.py \
     tests/test_validate_signal_db_smoke.py \
     -q
   ```

4. Run the impacted suite:

   ```bash
   .venv/bin/pytest \
     tests/test_trade_backtest.py \
     tests/test_signal_v2.py \
     tests/test_validate_signal_semantics.py \
     tests/test_validate_ultra_signal_production.py \
     tests/test_validate_signal_db_smoke.py \
     -q
   ```

5. Query the latest local market timestamp from `alt_core.market_1m`.
6. Resolve the package window:

   ```text
   end_at = min(floor(now_utc, hour), floor(max(market_1m.ts), hour)) - 24h
   start_at = end_at - window_days
   ```

   If `--end-at` is provided, use it directly and record `end_at_policy = "manual_override"`.

7. Run the real DB smoke test:

   ```bash
   ACTS_RUN_DB_SMOKE=1 .venv/bin/pytest tests/test_validate_signal_db_smoke.py -q
   ```

8. For each selector, run the validator using the fixed `end_at` and configured `window_days`.
9. Verify each selector artifact contains `summary.json`, `metadata.json`, `signals.csv`, and `README.md`.
10. Extract each selector's key fields:
    - `coverage_status`
    - `primary_label_complete_count`
    - `incomplete_label_count`
    - `signal_count`
    - `precision_before_dd8`
    - `avg_abs_mae_24h_pct`
    - `rule_version`
    - `feature_preparation_version`
    - `market_1m_timestamp_semantics`
    - `forward_scan_start_policy`
11. If `--comparison-root` contains traceable baseline/candidate config, run comparison mode and store `comparison.json` plus `comparison_README.md` under `comparisons/`.
12. If no traceable comparison config exists, record:

    ```text
    comparison_not_run: missing_traceable_baseline_candidate_config
    ```

13. Write final `run_manifest.json` and `EVIDENCE_PACKAGE.md`.

## Run Manifest

`run_manifest.json` is the machine-readable audit entrypoint. It must include at least:

```json
{
  "run_started_at": "2026-04-25T00:00:00+00:00",
  "run_finished_at": "2026-04-25T00:00:00+00:00",
  "git_sha": "abc123",
  "worktree_dirty": true,
  "window_days": 30,
  "end_at_policy": "db_aware_max_market_ts_minus_24h",
  "end_at": "2026-04-24T00:00:00+00:00",
  "latest_market_1m_ts": "2026-04-25T00:00:00+00:00",
  "selectors": [],
  "tests": [],
  "db_smoke": {},
  "selector_artifacts": {},
  "comparison": {},
  "overall_status": "passed_with_diagnostics"
}
```

The exact timestamps and selector maps are runtime values, but the keys above are required.

## Evidence Package README

`EVIDENCE_PACKAGE.md` is the human-readable review entrypoint. It must use these sections:

```text
# Validation Evidence Package

## Gate Summary
## Test Results
## DB Smoke
## Window
## Selector Artifacts
## Comparison
## Evidence Decision
## Caveats
```

The selector table must distinguish artifact availability from evidence quality. A selector with complete files but `coverage_status != trusted` is a valid diagnostic artifact, not threshold evidence.

The comparison section must state whether comparison ran. If it did not run, the reason must be explicit and must not be phrased as support for any threshold change.

## Status Model

The runner should classify outcomes with a small status vocabulary:

```text
evidence_eligible
diagnostic_only
comparison_not_run
gate_failed
passed
passed_with_diagnostics
failed
```

Selector-level evidence is `evidence_eligible` only when required files exist and `coverage_status == trusted`. Otherwise, selector output is `diagnostic_only`.

Package-level success requires:

- Targeted tests pass, unless `--skip-tests` was used. A skipped test phase prevents a passing evidence gate.
- Impacted tests pass, unless `--skip-tests` was used.
- `ACTS_RUN_DB_SMOKE=1` executes rather than skipping.
- All expected selector artifacts exist.
- Required selector fields can be read from `summary.json` and `metadata.json`.
- Comparison either produces a concrete comparison artifact or records `comparison_not_run` with a traceable reason.

If all gates pass but one or more selectors are diagnostic-only, the package status is `passed_with_diagnostics`.

## Error Handling

Hard failures should exit non-zero but still write a partial manifest when possible. Hard failures include:

- Targeted tests fail.
- Impacted tests fail.
- DB latest timestamp cannot be queried and no manual `--end-at` is provided.
- `ACTS_RUN_DB_SMOKE=1` skips or fails.
- A validator subprocess exits non-zero.
- A selector artifact is missing `summary.json`, `metadata.json`, `signals.csv`, or `README.md`.
- Required fields cannot be parsed from a selector artifact.

Diagnostic-only conditions should not necessarily fail the runner, but they must prevent evidence-backed conclusions:

- `coverage_status != trusted`.
- Sample counts are too small for threshold evidence.
- Comparison config is missing.
- Comparison result is `not_supported`, `experimental_only`, or `insufficient`.

Evidence-backed threshold claims are allowed only when comparison artifacts themselves report trusted 30-day evidence and any required trusted 90-day review. The runner must not infer this from raw selector artifacts alone.

## Testing Strategy

Normal automated tests must not depend on a real database. Add runner tests with monkeypatched subprocess and DB-query helpers to cover:

- Default selector expansion.
- DB-aware `end_at` calculation.
- Manual `--end-at` override.
- Test command logging.
- Real DB smoke command classification, including skip-as-failure behavior for the evidence gate.
- Selector artifact discovery and required-file checks.
- Required metric extraction.
- Missing comparison config handling.
- Partial manifest writing on hard failure.
- `EVIDENCE_PACKAGE.md` section generation.

The existing `ACTS_RUN_DB_SMOKE=1` test remains environment-gated. The runner treats it as mandatory during a formal evidence package run, while ordinary pytest remains DB-independent.

## Acceptance Criteria

- A formal run creates `artifacts/autoresearch/validation/<YYYY-MM-DD>/`.
- The package contains `EVIDENCE_PACKAGE.md` and `run_manifest.json`.
- Test commands, exit codes, and log paths are recorded.
- The DB-aware `end_at` and latest DB timestamp are recorded.
- `ACTS_RUN_DB_SMOKE=1` is run and classified as executed, skipped, or failed.
- All eight default selectors produce 30-day artifacts.
- Each selector artifact exposes required coverage, sample, and metric fields in the top-level package summary.
- Missing comparison config is recorded as `comparison_not_run: missing_traceable_baseline_candidate_config`.
- Comparison is run only from traceable config files and never synthesized by the runner.
- The final README separates `evidence_eligible`, `diagnostic_only`, `comparison_not_run`, and `gate_failed` outcomes.
- No threshold change is described as production-ready unless comparison artifacts provide trusted 30-day evidence and any required trusted 90-day review.

## Follow-Up Work

After the P0 evidence package is working, open separate specs or plans for:

- Data-health metadata such as latest benchmark timestamps, forward coverage rates, and material gap summaries.
- No-lookahead regression coverage for prepared features and selector decisions.
- Seven-day shadow validation rollout.
- Batch forward-row fetching for performance once trust semantics are stable.
